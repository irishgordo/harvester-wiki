We use GitHub Projects to manage development, QA, and community issue tracking. These are based on two-week sprints and follow the same sprint time window.

* [Harvester Sprint](https://github.com/orgs/harvester/projects/7?query=sort%3Aupdated-desc+is%3Aopen)
* [QA Sprint](https://github.com/orgs/harvester/projects/20)
* [Community Sprint](https://github.com/orgs/harvester/projects/10?query=sort%3Aupdated-desc+is%3Aopen)

## Harvester Sprint

RD members should ensure the following actions:

* Plan issues for the next sprint by adding them to the backlog of the [next sprint](https://github.com/orgs/harvester/projects/7/views/9) by the end of the current sprint.
* Move issues to ready-for-testing status by the end of the current sprint. For details on each status, please refer to [[Issue Management]].

## QA Sprint

QA members should ensure the following actions:

* Plan issues for the next sprint by adding them to the backlog of the [next sprint](https://github.com/orgs/harvester/projects/20/views/4) by the end of the current sprint.
* Move issues to ready-for-testing status by the end of the current sprint. 
* Verify ready-for-testing issues from the previous Harvester Sprint before addressing QA issues in the current sprint. Verifying ready-for-testing issues by the release schedule is the highest priority.

## Community Sprint

Community Coorinator should ensure the following actions:

* Check issues in the [current sprint](https://github.com/orgs/harvester/projects/10/views/9).
* Move issues to a corresponding status when working with reporters.
* Move issues to resolved status if they have been confirmed and decided to proceed. By default, the issue will remain open and be automatically added to the planning milestone.
* Move issues to closed status if they have been confirmed not to proceed, with reasons such as won't fix, invalid, or duplicate. The issue will then be closed.
